## 三、模块B：多Agent核保引擎（✅ 已部署 —— 实际实现复盘）

### 3.1 实际工作流拓扑

```
Start(INSURE_inquiry)
  │
  ▼
LLM1:信息提取 ─── 提取结构化字段 (temperature=0.7, structured_output=on)
  │
  ▼
代码执行:查询构造器 ─── 清洗think标签 + 构造精准query + 传递user_data
  │
  ▼
知识检索 ─── Top K=8, 混合检索, qwen3-rerank重排序
  │
  ├──────────────────────────────────┐
  ▼                                  ▼
LLM2:职业风险评估                    LLM3:健康风险评估
  │ (context=知识检索)                │ (context=知识检索)
  │ System Prompt内含降级规则        │ System Prompt内含降级规则
  │                                  │
  └────────────┬─────────────────────┘
               ▼
代码执行:定价计算 ─── safe_parse ×3 → 费率查表 → 风险系数计算 → JSON输出
  │
  ▼
LLM4:决策汇总 ─── 综合评估 → 四档决策 (temperature=0.7)
  │
  ▼
LLM5:报告生成 ─── Markdown格式化输出 (temperature=0.7)
  │
  ▼
End(OUTPUT)
```

### 3.2 设计 vs 实际：关键差异

以下是从 YML 导出文件对比原始设计文档发现的差异，每个差异都是踩坑后的真实调整：

| # | 配置项 | 原始设计 | 实际部署 | 原因/影响 |
|---|--------|----------|----------|-----------|
| 1 | **Start 变量** | `query` (paragraph, 5000字) | `INSURE_inquiry` (text-input, 256字) | 限制输入长度避免超长上下文导致后续节点处理缓慢；text-input 类型更简洁 |
| 2 | **Agent2/3 System Prompt** | 仅写"从知识库检索结果中匹配" | 内嵌完整降级规则表（职业6级分类、健康9类加费标准） | **核心优化**：当KB检索不到精确匹配时，Agent仍有规则可用，不会"乱编"——这是"KB优先 + 内置兜底"的双层架构 |
| 3 | **Top K** | 6 | 8 | 多风险因子叠加时（职业+健康+年龄+吸烟+BMI），6条片段不够覆盖所有维度 |
| 4 | **Temperature** | LLM1=0.3, 其余未指定 | 全部 0.7 | 提取节点用 0.7 也没出现不稳定（因为有 structured_output + retry），但理论上 0.3 更稳——这是一个可优化点 |
| 5 | **定价代码-保额解析** | 简单取数字 | 增加"万"单位检测，若含"万"则 ×10000 | 用户输入"50万"时解析为 500000 而非 50 |
| 6 | **节点命名** | `Agent1:信息提取` | `LLM1:信息提取` | Dify 中 LLM 节点默认命名风格，不影响功能但面试时要说清楚 |
| 7 | **报告生成变量引用** | 全部用双花括号 `{{}}` | 部分用了三花括号 `{{{}}}` | **潜在问题**：三花括号在 Dify 中可能不做 HTML 转义，需验证是否影响报告输出 |

### 3.3 从模块B抽象出的通用设计模式

这些模式可以直接应用到模块A和模块C的设计中：

**模式1：「KB优先 + 内置兜底」双层规则架构**
```
应用场景：任何需要 LLM 基于规则做判断的节点
做法：
  Layer 1 - KB检索：提供详细的、可热更新的领域规则
  Layer 2 - System Prompt：嵌入核心规则的降级版本（KB无结果时使用）
效果：避免 RAG 检索失败时 LLM 胡编乱造
```

**模式2：「提取→构造→检索」三段式查询链路**
```
应用场景：需要从自然语言中提取信息再检索知识库
做法：
  Step 1: LLM提取结构化字段（JSON输出）
  Step 2: Code节点构造精准查询语句
  Step 3: KB节点用精准查询做检索
效果：检索召回率大幅提升（模块B从几乎为0到稳定可用）
```

**模式3：「safe_parse 防御性解析」**
```
应用场景：任何接收 LLM 输出的 Code 节点
做法：
  1. 清洗 <think> 标签
  2. 提取第一个 JSON 对象（{ 到 }）
  3. try/except 兜底返回空字典
  4. 所有字段提供默认值
效果：系统不因 LLM 输出波动而崩溃
```

**模式4：「并行独立评估 + 汇聚计算」**
```
应用场景：需要多维度独立评估后汇总的场景
做法：
  1. 从KB检索节点引出并行分支
  2. 每个分支独立评估一个维度
  3. 所有分支汇入一个 Code 节点做融合计算
效果：延迟减半，每个维度评估质量提高
```

## 四、模块A：智能产品配置助手（v2.0 深度重写）

### 4.1 设计思路

原始设计过于简略——只有一个 5 步流程图和输出 JSON 示例。模块B 的经验表明，实际搭建需要考虑：输入验证、查询构造、KB兜底、输出格式校验、异常处理。模块A 同样需要这些。

**核心差异**：模块A 不是"对话"而是"配置生成"——用户描述需求 → 系统生成完整的产品定义。这更像一个**表单替代品**而非客服机器人。因此选用 **Workflow** 而非 Chatflow。

### 4.2 实际可搭建的工作流

```
Start(product_requirement)  ← 用户用自然语言描述要配置的产品
  │
  ▼
LLM_A1:需求解析 ─── 提取7个产品要素为结构化JSON (temperature=0.3)
  │                 ┌─ product_type (险种)
  │                 ├─ target_customers (客群: 年龄/收入/职业)
  │                 ├─ coverage_scope (保障范围)
  │                 ├─ sum_insured_range (保额区间)
  │                 ├─ payment_methods (缴费方式)
  │                 ├─ special_features (特色条款: 豁免/分红/多次赔付)
  │                 └─ competitor_reference (参考竞品，如有)
  │
  ▼
Code_A1:查询构造 ─── 应用模式2「提取→构造→检索」
  │                  构造3条并行查询：
  │                  Q1: 监管要求 + 合规边界（该险种的监管规定）
  │                  Q2: 同类产品参考（竞品参数/市场惯例）
  │                  Q3: 条款模板（标准条款/免责清单）
  │
  ▼
KB:产品知识库 ─── 检索：监管文件/同业产品/条款模板
  │               Top K=6, 混合检索
  │
  ├──────────────────────────────────────────┐
  ▼                                          ▼
LLM_A2:参数推导                              LLM_A3:合规检查
  │ 应用模式1「KB优先+内置兜底」               │ 应用模式1
  │ 推导：等待期/免责条款/费率框架/            │ 检查：该产品是否违反监管
  │       缴费周期/最低保额/最高保额            │       是否有必须包含的条款
  │ System Prompt内含通用产品参数表           │       是否有禁止使用的费率结构
  │                                          │
  └────────────┬─────────────────────────────┘
               ▼
Code_A2:产品定义生成 ─── 应用模式3「safe_parse」
  │                      1. 接收 LLM_A2 参数 + LLM_A3 合规结论
  │                      2. 验证字段完整性（哪些必填字段缺失？）
  │                      3. 检查数值合理性（保额范围是否合理？费率是否异常？）
  │                      4. 生成 InsureMO 兼容的 JSON
  │                      5. 若有合规风险，标记 risk_flags
  │
  ▼
LLM_A4:产品说明生成 ─── 将JSON转为人类可读的产品说明书 (Markdown)
  │                     含：产品概述/保障详情/费率框架/适合人群/注意事项
  │
  ▼
End(product_definition_json + product_brochure_md + compliance_report)
```

### 4.3 关键节点详细设计

**LLM_A1:需求解析 —— System Prompt 核心片段**：
```text
你是一个保险产品需求分析师。从用户自然语言描述中提取产品配置要素。

提取字段：
- product_type: 重疾险/医疗险/定期寿险/意外险/年金险/两全险
- target_age_range: {min, max}
- target_income_range: {min, max, unit: "万"}
- target_occupations: 职业类别列表
- coverage_scope: 保障责任清单（如"120种重疾""轻症30%给付"）
- sum_insured_range: {min, max, step}
- payment_period: ["趸交","5年","10年","20年","30年"]
- payment_frequency: ["月缴","季缴","半年缴","年缴"]
- special_clauses: 特色条款（轻症豁免/重症多次赔付/身故返还/分红）
- waiting_period_days: 等待期（未提及则填null，不要编造）
- competitor_reference: 用户提到的参考产品名称
- regulatory_region: 销售地区（影响监管适用规则）

输出合法JSON。未提及字段填null，不编造。
```

**Code_A2:产品定义生成 —— 核心校验逻辑**：
```python
# 伪代码结构
def main(params_json, compliance_json, user_data):
    params = safe_parse(params_json)
    compliance = safe_parse(compliance_json)
    
    # 字段完整性校验
    required_fields = ["product_type", "sum_insured_range", "coverage_scope"]
    missing = [f for f in required_fields if not params.get(f)]
    
    # 数值合理性校验
    if params.get("waiting_period_days"):
        if params["waiting_period_days"] < 30:
            flag("等待期过短，需确认是否符合监管要求")
    if params.get("sum_insured_range", {}).get("max", 0) > 5000000:
        flag("保额上限超过500万，需额外风控审批")
    
    # 合规风险标记
    risks = []
    if compliance.get("has_regulatory_issues"):
        risks = compliance["issues"]
    
    # 生成最终JSON
    return {
        "product_definition": {...},
        "validation": {"missing_fields": missing, "warnings": [...]},
        "compliance_risks": risks,
        "status": "draft" if missing else "ready_for_review"
    }
```

## 五、模块C：智能保单管家（v2.0 深度重写）

### 5.1 设计思路

原始设计只给了一个简单的条件分支图。实际实现需要处理三个难点：
1. **意图歧义**：用户可能一句话混合多个意图（"我的保单还有效吗？另外如果我查出结节能赔吗？"）
2. **上下文缺失**：用户可能不记得保单号，需要从对话中推断
3. **多轮对话**：不是一问一答就结束，可能需要追问

因此模块C 选用 **Chatflow**（而非 Workflow），需要用到 Dify 的 Conversation Variables 来维护多轮上下文。

### 5.2 Chatflow 设计

```
用户输入 ──→ LLM_C1:意图识别+信息提取 ──→ 条件路由
                      │
                      │ 提取：intent, policy_type, health_condition, claim_reason
                      │
          ┌───────────┼───────────┐
          ▼           ▼           ▼
    [保单查询]   [保障咨询]   [理赔问询]
          │           │           │
          ▼           ▼           ▼
    KB检索:       KB检索:       KB检索:
    保单条款      保障范围       理赔规则
    缴费记录      免责条款       理赔流程
          │           │           │
          ▼           ▼           ▼
    LLM_C2a:      LLM_C2b:      LLM_C2c:
    保单解读      保障分析       理赔指引
    - 用大白话    - 这个病在不在  - 理赔步骤
      解释条款      保障范围      - 所需材料
    - 下次缴费    - 能赔多少     - 预计时效
      时间        - 有没有免赔额 - 注意事项
          │           │           │
          └───────────┼───────────┘
                      ▼
              LLM_C3:回复生成
              - 如果一句话含多个意图，逐条回复
              - 如果信息不足，追问具体信息
              - Markdown格式
                      │
                      ▼
                  回复输出
              (同时更新 Conversation Variables:
               last_intent, last_policy_type, last_health_condition)
```

### 5.3 多轮对话状态管理

模块C 最大的设计挑战是**跨轮次上下文维护**。用户可能先问保单内容，下一轮直接说"那如果我得了XX呢？"——这个"那"字依赖上一轮的上下文。

```yaml
# Dify Conversation Variables 设计
conversation_variables:
  - name: user_profile
    type: object
    description: 从对话中逐步收集的用户信息
    fields:
      - policy_holder_name: 投保人姓名
      - policy_number: 保单号（如有）
      - policy_type: 投保险种
      - coverage_amount: 保额
      - effective_date: 生效日期
      - health_history: 对话中提到的健康状况列表
      
  - name: conversation_stage
    type: string
    description: 当前对话所处阶段
    enum: [greeting, info_collection, query_handling, follow_up, closed]
    
  - name: last_intent
    type: string
    description: 上一轮的意图，用于处理省略式追问
```

### 5.4 意图识别详细设计

```text
LLM_C1 System Prompt 核心：

你是一个保险客服意图识别器。分析用户输入，输出JSON。

意图分类：
1. policy_lookup - 保单查询（"我的保单还有效吗""下次什么时候缴费"）
2. coverage_inquiry - 保障咨询（"这个病在保障范围吗""能赔多少"）
3. claim_guidance - 理赔指引（"怎么理赔""需要什么材料""去哪里办理"）
4. renewal_reminder - 续保相关（"快到期了怎么续""保费涨了吗"）
5. complaint - 投诉/不满（转人工）
6. mixed - 包含多个意图（逐条标注）

提取信息（只提取用户明确提到的）：
- health_condition: 提到的疾病/症状
- treatment: 提到的治疗方式（住院/手术/门诊）
- date_reference: 时间相关表述（"去年""上个月""下周"）
- urgency_level: 紧急程度（1-3，3为投诉或紧急理赔）

当前对话历史摘要：{{#conversation_history_summary#}}

如果用户用"那""这个""它"等代词指代上文内容，
请结合对话历史推断具体指代什么。
```

### 5.5 关键场景处理策略

**场景1：用户说"我这个情况能赔吗"但没有提供足够信息**
```
系统行为：
1. 从 Conversation Variables 检查是否已有健康信息
2. 如果没有，追问"请问您具体是什么情况呢？比如是什么疾病或症状？"
3. 追问时给出示例，降低用户输入门槛
```

**场景2：用户一句话混合"查询保单"和"问理赔"**
```
用户输入："我的重疾险保单还有效吗？另外我上个月查出甲状腺结节，这个能赔吗？"

系统行为：
1. LLM_C1 识别为 mixed 意图
2. 并行处理两个子意图
3. LLM_C3 回复时用 "---" 分隔两部分，结构清晰
```

**场景3：理赔指引 —— 用户不知道具体流程**
```
系统行为（LLM_C2c）：
1. 列出理赔步骤（1→2→3→4）
2. 每步标注所需材料和时间节点
3. 给出预计到账时间
4. 标注"以上为通用流程，具体以您保单条款为准"
```

### 5.6 模块C 知识库内容规划

| 文档 | 内容 | 用途 |
|------|------|------|
| 保单条款范本 | 不同险种的保障范围、免责条款、理赔条件 | 条款解读 |
| 理赔流程指南 | 各险种理赔步骤、所需材料清单、时效说明 | 理赔指引 |
| 常见拒赔案例分析 | 哪些情况不赔、为什么、如何避免 | 用户预期管理 |
| 续保规则 | 宽限期、复效条件、保费调整规则 | 续保咨询 |

## 六、统一对话入口（v2.0 新增）

### 6.1 设计思路

原始设计中统一入口只是一个架构图上的方块。实际实现需要一个**单独的 Chatflow 作为路由层**，根据用户意图分发到对应的子模块。

但由于 Dify 的限制（一个 Chatflow 不能直接调用另一个 Workflow/Chatflow），有两种实现路径：

**方案A（推荐）**：用一个 Chatflow 实现意图路由 + 内嵌三个模块的逻辑。模块B 的核心链路（信息提取 → 检索 → 评估 → 决策）作为 Chatflow 中的一个条件分支。

**方案B（更现实）**：保持三个独立 Dify 应用，在外部（如飞书机器人/网页）做第一层意图识别和路由。面试时讲清楚架构设计即可，Demo 展示时分别打开三个应用。

推荐采用**方案B**，原因：
- 模块B 已经是 Workflow 且运行稳定，拆进 Chatflow 会增加复杂度
- 面试时分别展示三个独立应用更能体现"模块化设计"思维
- 实际生产中也是微服务架构，各模块独立部署

### 6.2 方案B 的意图路由 Chatflow

```
用户输入 ──→ LLM_R1:意图识别 ──→ 条件分支
                │
                │ 输出: {intent: "underwriting"|"product_config"|"policy_service"|"general"}
                │
        ┌───────┼───────┬───────┐
        ▼       ▼       ▼       ▼
   [核保咨询][产品配置][保单服务][通用问答]
        │       │       │       │
        ▼       ▼       ▼       ▼
   提示用户   提示用户  提示用户  LLM回复
   "请打开    "请打开   "请打开  常见问题
    核保助手" 产品配置  保单管家"
             助手"            
```

> 面试时说："实际部署中，统一入口可以由前端或飞书机器人实现，根据 LLM 识别的意图路由到不同的 Dify 应用。这在架构上就是 API Gateway 模式的 AI 版本。"

## 九、面试叙事线（v2.0 更新）

> "我研究易保 InsureMO 的产品架构后，发现传统保险中台的三个痛点：配置门槛高、规则更新慢、决策不透明。
>
> 我用 Dify 搭建了 InsureFlow，**目前核心模块——多Agent核保引擎已经实际部署并跑通了 8 个测试用例。** 这个过程中我踩了不少坑，比如知识库检索一直返回空——根因是直接用自然语言做 Embedding 查询，口语化和结构化规则的语义距离太远。我在 LLM 提取结构化信息之后加了一层代码节点做查询构造，相当于在用户和知识库之间加了一层语义翻译。
>
> 另一个关键设计是System Prompt内嵌降级规则——当KB检索不到精确匹配时，Agent仍有兜底规则可以用，避免乱编。我把这个称为'KB优先+内置兜底'双层架构。
>
> 基于模块B的经验，我完成了模块A（智能产品配置助手）和模块C（智能保单管家）的实操级设计方案，可直接搭建。
>
> 技术上：RAG热更新、多Agent并行评估减少50%延迟、safe_parse防御性解析保证系统不崩溃、结构化输出保证系统间对接。"
