## v0.1 — 概念验证（2026-06-10）

### 做了什么
- 创建第一个 Dify Workflow：单 LLM 节点做"信息提取+职业评估+健康评估+决策"
- 写了超长 Prompt（约 2000 字），把所有核保规则塞进 System Prompt
- 没有知识库，没有代码节点，没有并行分支
- 测试了 2 个简单输入

### 发现的问题
1. **单 LLM 做所有事，准确率低**：职业评估和健康评估互相干扰，常出现"职业评估详细但健康评估只有一句话"
2. **输出格式不稳定**：有时输出 Markdown，有时输出纯文本，偶尔在 JSON 前后加解释语
3. **核保规则更新需要改 Prompt**：每改一条规则都要重新编辑长 Prompt
4. **响应慢**：单 LLM 处理 2000 字 Prompt + 用户输入 ≈ 6-8 秒

### 关键决策
> **结论**：单 LLM 架构不可行。需要拆分为多个专业化 Agent，并引入知识库做规则管理。

### 迭代方向确认
- [x] 拆分 Agent（下一版本最优先）
- [x] 引入 RAG 知识库
- [x] 加入 Code 节点做精确计算
- [ ] 并行化评估流程

## v0.2 — 首次可运行（2026-06-12）

### 做了什么
- **架构重构**：从单 LLM 拆分为 5 个 LLM 节点
  - LLM1: 信息提取
  - LLM2: 职业风险评估
  - LLM3: 健康风险评估
  - LLM4: 决策汇总
  - LLM5: 报告生成
- 创建 `核保规则知识库`，包含 8 类规则文档
- 引入 KB 知识检索节点
- 引入 Code 节点做定价计算
- 探索性测试 3 个用例

### 遇到的 3 个关键问题及修复

#### 问题 A：知识库检索返回空 ⭐⭐⭐⭐⭐（核心突破）
- **表现**：运行流程后 KB 检索节点返回 0 条结果
- **排查**：
  1. 确认文档已上传并索引完成 → ✓
  2. 在知识库页面直接搜索关键词 → 能召回
  3. 检查 KB 节点的查询变量 → 发现用的是 `{{#start.query#}}`（原始自然语言）
  4. **根因**：用户输入"我30岁，程序员，有高血压"是口语化表达，而知识库文档里是"3级风险职业 — 加费10%"这样的结构化规则文本。Embedding 语义匹配度远低于阈值，导致混合检索也召不回
- **修复**：新增 Code 节点「查询构造器」，放在 LLM1 之后、KB 检索之前
  1. 从 LLM1 的结构化输出中提取关键字段
  2. 拼接成精准关键词查询："职业：程序员，职业风险等级与加费比例；险种：重疾险，基准费率；健康状况：高血压，加费标准"
  3. KB 检索改用构造后的查询
- **效果**：检索召回率从 ~0% 提升至稳定召回相关片段（Top K=8 中至少 5-6 条相关）

#### 问题 B：Agent 看不到知识库内容
- **表现**：Agent2/Agent3 的评估结果与知识库规则不一致，像是"自己编的"
- **根因**：Dify 中 LLM 节点不会自动获取上游 KB 检索结果，需要在每个 LLM 节点的"上下文"配置中显式指定来源节点
- **修复**：在 Agent2 和 Agent3 的节点配置中，将上下文设置为 KB 检索节点，User Prompt 中使用 `{{#context#}}` 引用
- **效果**：Agent 输出与知识库规则一致

#### 问题 C：并行评估未生效
- **表现**：Agent2 和 Agent3 虽然在画布上从 KB 检索节点并行引出，但实际执行顺序不对
- **根因**：Dify 工作流中，同一节点的多个输出分支需要正确的连线方式才能触发自动并行
- **修复**：确保两个 Agent 都直接从 KB 检索节点的输出连线，不经过其他中间节点
- **效果**：并行生效，总延迟从 ~16 秒降至 ~10 秒

### 迭代方向确认
- [ ] Prompt 精调（每项评估的准确性）
- [x] 加入 safe_parse 容错（下一版本关键）
- [x] 补齐降级规则
- [ ] 测试用例体系建设

## v1.0 — MVP 基线（2026-06-14）

### 做了什么
- **safe_parse 防御性解析**：在所有 Code 节点加入统一的 JSON 安全解析函数
  - 正则清洗 `<think>...</think>` 标签
  - 提取第一个完整 JSON 对象
  - try/except 兜底返回空字典
  - 所有字段提供默认值
- **降级规则嵌入**：Agent2/Agent3 的 System Prompt 中内置核心降级规则
  - 当 KB 检索无结果时，Agent 仍有规则可用
  - "KB优先 + 内置兜底"的双层架构成型
- **定价计算精确化**：
  - 4 套险种费率表写入 Code 节点
  - 增加"万"单位检测（用户输入"50万"→解析为 500000）
  - 保额上限校验
- **Top K 从 6 调整到 8**：多风险因子叠加时，6 条片段覆盖不足
- **撰写 8 个系统化测试用例**，覆盖全部 4 档决策

## v1.1 — 稳定性增强（2026-06-16）

### 做了什么

#### 改动 1：新增边缘用例测试
- 新增 TC09（信息极少）、TC10（信息矛盾）
- TC09 发现：早期版本在全字段 null 时会崩溃。修复：所有 Code 节点增加 null 默认值
- TC10 验证通过：矛盾信息不导致系统混淆

#### 改动 2：LLM 输出容错加固
- 在 LLM 节点配置中开启结构化输出（structured_output: on）
- 开启重试机制（最大 3 次，间隔 1000ms）
- 在 Assistant Prefill 中预填 `{` 引导 JSON 输出

#### 改动 3：输入变量约束
- Start 节点的 `INSURE_inquiry` 从 paragraph 改为 text-input，限制 256 字
- 原因：避免超长上下文导致后续节点处理过慢
- 权衡：牺牲了长文本输入能力，但保证了稳定性和响应速度

#### 改动 4：费率表微调
- 基准保费表增加了女性费率（性别系数 0.95）
- 年龄系数区间细化：原 `>65(2.0)` 改为 `>65(拒保)`
- 吸烟系数增加"戒烟时间"维度（戒烟1年以上的加费减半）

#### 改动 5：知识库文档优化
- 核保规则文档分段从自动分段改为手动标注
- 每个规则段落添加明确的关键词标签（如 `[职业分类] [3级风险] [外卖骑手]`）
- 效果：检索召回率进一步提升

### 测试结果
- 回归测试 TC01-TC08：全部通过
- 新增 TC09-TC10：全部通过
- 新增鲁棒性专项测试 8 项：7/8 通过

## v2.1 — API级联调试与HTTP节点兼容性排查（2026-06-17）

### 背景

在 v2.0 中完成了三模块联动方案设计，包括两套实现：
- **Scheme A**（Hub统一智能中枢）：6节点 Workflow，路由逻辑嵌入 LLM Prompt，✅ 已导入并验证通过
- **Scheme B**（API级联自动串联）：11节点 Workflow，通过 HTTP Request 节点串行调用三个模块的 Workflow API

Scheme B 的 YML 文件已创建，但在 Dify 中导入后出现「一直转圈、没有任何节点开始执行」的问题。

### 遇到的 6 个问题及排查过程

#### 问题 1：导入后首次运行一直转圈 ⭐⭐⭐⭐⭐（核心问题，耗时最长）

- **表现**：导入 `智保通-全链路API自动串联.yml`，配置环境变量并发布后，输入测试用例点击发送，页面一直显示转圈动画，**所有节点均无状态显示**（连 Start 节点都没有执行标记）
- **对比验证**：Hub（统一智能中枢，同模型同平台）输入"你好"能正常返回 → 确认 DeepSeek 模型服务正常
- **第一轮排查**：
  1. 检查环境变量配置无误（DIFY_BASE_URL + 3个API Key均已填写）
  2. 重新打开两个 LLM 节点，手动重新点选 `deepseek-v4-pro` 模型 → 保存 → 发布 → 仍转圈
  3. 检查 Code 节点的变量引用是否正确关联 → 均正确 → 仍转圈
  4. 临时创建 Start→End 直连测试 → 仍转圈
- **第一轮结论**：问题不在模型绑定或变量引用，可能是 YML 文件本身存在 Dify 无法处理的配置

#### 问题 2：YML 结构兼容性排查

- **排查方向**：对比已通过验证的 Hub YML 和 API级联 YML 的差异
- **发现的差异项**：
  1. `reasoning_format: tagged` — Hub 也有且工作正常，排除
  2. `structured_output_enabled: true` — Hub 也有且工作正常，排除
  3. `value_type: secret` → API Key 环境变量使用了 secret 类型，Hub 无环境变量
  4. `features.opening_statement` 和 `suggested_questions` 有复杂内容
  5. 节点数多（11 vs 6），变量引用链路长
- **第一次重写**：基于 Hub 的已验证结构为模板，做了以下修改：
  - 删除 `reasoning_format: tagged`
  - 关闭 `structured_output_enabled`
  - 环境变量 `value_type` 从 `secret` 改为 `string`
  - 清空 `opening_statement` 和 `suggested_questions`
  - 简化节点 ID 格式
- **结果**：重写后导入仍转圈

#### 问题 3：创建调试版本逐步隔离问题 ⭐⭐⭐⭐

- **思路**：既然完整版（11节点）转圈，创建最小化版本来定位问题层级
- **Debug V1**（3节点：Start → LLM → End）：
  - 导入 → 配置模型 → 发布 → 输入"测试"
  - **结果：✅ 通过**，返回"调试成功：工作流基础结构正常运行"
  - **结论**：Dify 基础结构没问题，LLM 节点可正常执行
- **Debug V2**（6节点：Start → LLM → Code → HTTP_A → Code → End）：
  - 仅添加 1 个 HTTP Request 节点 + 配套 Code 节点
  - 仅需 2 个环境变量（DIFY_BASE_URL + API_KEY_MODULE_A）
  - **结果：❌ 转圈**，同完整版症状
- **关键结论**：**问题锁定在 HTTP Request 节点**。只要工作流中包含 HTTP Request 节点，Dify 就无法启动执行

#### 问题 4：排查是否为 URL 拼接变量导致

- **假设**：`{{#env.DIFY_BASE_URL#}}/v1/workflows/run` 这种 env var + 字符串拼接的写法，可能 Dify 0.6.0 不支持或解析失败导致初始化卡死
- **Debug V3**（同 V2 结构，但 URL 硬编码）：
  - URL 直接写死为 `https://cloud.dify.ai/v1/workflows/run`（不拼接变量）
  - 仅保留 `API_KEY_MODULE_A` 一个 env var（用于 Bearer Token）
  - **状态**：🔲 待测试
- **如果 V3 通过** → 根因是 env var 在 URL 字段中的字符串拼接不被支持
- **如果 V3 也转圈** → 根因更深层，可能是：
  - Dify Cloud 的沙箱环境禁止 HTTP Request 节点发起外部请求
  - Dify 0.6.0 的 HTTP Request 节点本身存在 bug
  - Dify Cloud 免费版限制 HTTP 节点使用

#### 问题 5：API Key 创建方式确认

- **用户疑问**：三个模块的 API Key 是在各模块内各创建一个，还是在一个页面创建三个？
- **结论**：在三个模块各自的「API 访问」页面各创建一个 Key，共三个，分别粘贴到环境变量。操作方式正确。
- **验证方法**：每个模块的 API 访问页面显示的 API Key 均以 `app-` 开头，格式一致

#### 问题 6：环境变量类型选择

- **原设计**：`API_KEY_MODULE_B` 和 `API_KEY_MODULE_C` 使用 `value_type: secret`（加密存储）
- **排查发现**：secret 类型变量在 YML 导入后的初始化流程可能与 string 类型不同，增加初始化复杂度
- **修复**：全部改为 `value_type: string`，降低初始化时的变量解析负担
- **安全性权衡**：生产环境应使用 secret，但当前 Demo 阶段优先保证可用性

### 关键认知

> 这次排查让我深刻体会到：AI 产品开发中，**平台本身的限制和兼容性**是一等公民问题，和 Prompt 设计、架构选型同等重要。「能跑通」和「能导入」之间隔着的不是代码逻辑，而是对平台底层行为的理解。
>
> Dify 作为一个低代码平台，隐藏了大量实现细节。当遇到问题时，传统的"读日志→定位→修复"路径走不通（因为没有日志可读），必须依赖**控制变量法逐层排除**。这种调试方法论本身，就是"工程化能力"的体现。

## 迭代教训总结

### 做对了什么
1. **先单LLM验证可行性，再拆分Agent**——避免过早优化
2. **问题驱动迭代**：每次版本都是解决上一版本的实际问题，不是凭空设计
3. **核心模块先跑通，再扩展其他模块**：模块B是基础，A和C是延伸
4. **每个技术决策都有记录**：方便面试时解释"为什么这么做"

### 做错了什么（坦诚版）
1. **v0.1 的 Prompt 写得太长**：2000 字的 Prompt 导致 LLM 遵循度下降。教训：每个 Agent 的 Prompt 不超过 300 字
2. **一开始没做 safe_parse**：v0.2 测试时 Code 节点频繁报错才意识到 LLM 输出不是 100% 可靠的。教训：永远不假设 LLM 输出格式稳定
3. **知识库分段用了自动分段**：没有手动优化，导致检索效果不够好。v1.1 改为手动标注后才明显改善
4. **测试用例写得太晚**：v1.0 才开始系统化测试，应该在 v0.2 就做
5. **低估了 Dify 平台的配置复杂性**：上下文配置、变量引用、并行连线——这些都是需要实际踩坑才知道的细节
6. **低估了 Dify 平台的功能限制（v2.1 新增）**：在本地设计完整的 API 级联方案，生成了可导入的 YML，但未预先验证 Dify HTTP Request 节点在目标平台上的实际可用性。教训：涉及平台特有功能的方案，应该先做最小可行性验证（一个 HTTP 节点能否跑通），再投入完整方案设计和 YML 编写

### 如果重来一次
1. Day 1 就写测试用例，驱动开发
2. Day 1 就实现 safe_parse，而不是等到频繁报错才加
3. 先用 3 个简单用例验证链路，再扩展到 12 个用例
4. 不追求一步到位的完美架构，而是快速出 MVP 再迭代

>
> v1.0 我做了两件事：一是实现了 safe_parse 防御性解析，让 LLM 输出不稳定时系统不会崩溃；二是内嵌了降级规则，当 KB 检索失败时 Agent 仍有兜底规则可用。这是'KB 优先 + 内置兜底'的双层架构。
>
> v1.1 做了稳定性加固和测试用例扩展。v2.0 我把模块B的经验抽象成 4 个通用设计模式，推广到了模块A和模块C，并完成了三模块联动方案——包括嵌入式路由（Scheme A）和 API 级联编排（Scheme B）两套方案。
>
> v2.1 是 API 级联方案的调试版本。我遇到了一个有意思的问题：导入的 YML 在 Dify 中一直转圈，所有节点都不执行。因为 Dify 作为低代码平台没有暴露执行日志，我用了控制变量法逐层排查——先创建 3 节点的最小版本验证基础结构，再增量添加 HTTP 节点定位问题，最后通过硬编码 URL 隔离变量拼接的嫌疑。最终锁定了 HTTP Request 节点在 Dify Cloud 上的兼容性问题。这个过程让我深刻理解到：做 AI 产品，平台本身的限制和兼容性是一等公民问题，和 Prompt 设计、架构选型同等重要。

## v2.1 最终结论（2026-06-24 补充）

### Debug V3 测试结果

2026-06-24 完成 Debug V3 验证：HTTP 节点 URL 硬编码为 `https://cloud.dify.ai/v1/workflows/run`，仅保留 `API_KEY_MODULE_A` 环境变量。导入→配置→发布→测试，结果仍是长时间转圈无响应。

### 三级排查完整链路

| 版本 | 节点结构 | URL方式 | 结果 | 排除因素 |
|------|---------|---------|:---:|----------|
| Debug V1 | Start→LLM→End（无HTTP） | — | ✅ 通过 | — |
| Debug V2 | Start→LLM→Code→HTTP→Code→End | env变量拼接 | ❌ 转圈 | — |
| Debug V3 | 同V2 | 硬编码 | ❌ 转圈 | URL拼接不是原因 |

### 最终根因

**Dify Cloud 沙箱环境禁止 HTTP Request 节点发起外部 API 调用。** 这不是 YML 配置格式问题，不是 URL 变量拼接问题，而是平台层面的功能限制。

### 最终方案：A+B 组合

| 方案 | 实现 | 平台 | 状态 | 用途 |
|------|------|------|:---:|------|
| Scheme A (Hub) | `智保通-统一智能中枢.yml` | Dify Cloud | ✅ 已部署 | 日常演示 |
| Scheme B (编排) | `insureflow_orchestrator.py` | Python本地 | ✅ 可用 | API级联、展示架构能力 |

Python 脚本通过标准库直接调用三个模块的 Workflow API，实现 Start→模块A→解析→模块B→解析→模块C→汇总的完整链路，输出结构化全链路报告。

### 面试话术（"API级联最后怎么解决的？"）

> "排查到最后，我用了三级控制变量法确认了根因——Dify Cloud 的沙箱环境禁止 HTTP Request 节点出站。然后我写了一个 Python 编排脚本替代 Dify 内置的 HTTP 节点，用标准库直接调用三个模块的 Workflow API。这本质上是一个轻量级的 API Gateway + Service Orchestrator——每个模块独立部署，编排层负责数据适配和流程控制。这段经历让我体会到：低代码平台有它的能力边界，知道什么时候该跳出平台自己造工具，是 AI 产品经理很重要的判断力。"
