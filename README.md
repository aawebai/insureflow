# Dify智能核保助手 —— AI产品实践项目

## 这是什么

一个基于Dify搭建的智能核保助手。用户用自然语言描述个人信息（年龄、职业、健康状况等），系统自动评估风险、计算保费、输出核保结论和推理过程。

**这是一个学习型项目**，目的是通过动手实践理解AI产品从需求分析到交付测试的完整流程。不是生产级产品，是一个经过测试验证的Demo。

## 做了什么

### 核心功能

1. **自然语言输入** → 自动提取年龄、职业、健康状况、险种、保额等结构化信息
2. **知识库检索** → 根据用户信息从核保规则库中检索对应规则
3. **多维度风险评估** → 职业风险和健康风险并行评估，各自输出风险等级和加费比例
4. **保费计算** → 代码实现精确的保费公式计算（不让AI碰数字）
5. **核保报告** → 结构化输出结论、风险明细、保费明细、决策依据

### 工作流

用户输入 → 信息提取 → 查询构造 → 知识库检索 → [职业风险评估 ∥ 健康风险评估] → 定价计算 → 决策汇总 → 报告输出

### 核保结论分四档

- **标准体**：正常费率承保
- **次标准体**：加费承保
- **延期体**：建议观察后重新投保
- **拒保体**：拒绝承保

## 搭建过程中解决的实际问题

### 问题1：知识库检索返回空

- **现象**：搭好工作流第一次跑，知识库检索返回0条结果
- **排查**：确认文档已上传→直接搜关键词能搜到→检查发现检索用的是用户原始口语输入
- **根因**：用户口语（"血压有点高"）和知识库规则文本（"高血压1级加费20%"）语义差距大，向量搜索匹配不上
- **方案**：在用户输入和知识库检索之间加了一个"查询构造"步骤——先用LLM提取结构化字段，再用代码拼成精确查询语句
- **效果**：检索召回率从几乎为0提升到稳定可用

### 问题2：LLM输出格式不稳定

- **现象**：代码节点解析LLM输出的JSON时频繁报错
- **排查**：查看日志发现LLM输出带`<think>`标签，JSON前后有多余文字
- **根因**：DeepSeek模型会输出内部推理过程，偶尔不遵守严格的JSON格式要求
- **方案**：在所有接收LLM输出的代码节点加入容错函数——清洗标签→提取JSON→默认值兜底
- **效果**：系统鲁棒性大幅提升，异常输入不崩溃

### 问题3：串行流程响应慢

- **现象**：用户输入后要等约16秒才看到结果
- **分析**：职业评估和健康评估是独立的，但串行执行互相等待
- **方案**：改为并行分支，两个评估同时进行
- **效果**：延迟降至约8-10秒，减少约50%
