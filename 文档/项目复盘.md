## 项目概述

我用 Dify 搭了一个智能核保助手。用户用自然语言描述自己的情况（年龄、职业、健康状况等），系统自动评估风险、计算保费、输出核保结论。

项目定位是**学习型实践**——目的是通过动手做，理解从业务分析到需求拆解再到测试验证的完整过程。不是生产级产品。

---

## 一、为什么做这个项目

我对保险科技感兴趣，想通过动手实践来理解这个领域。选择核保场景是因为：

- 核保规则明确、流程清晰，适合做需求分析和流程拆解的练习
- 它不是一个简单的"聊天"，需要多步骤的逻辑判断和数据流转
- 保险相关的公开资料比较多，可以自学基础知识

在做的过程中，我经历了从业务学习→流程拆解→搭建测试→问题排查的完整过程，这恰好是 BA/需求分析岗位的核心工作内容。

### 技术选型

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 搭建平台 | Dify | 低代码平台快速验证想法，不需要从零写工作流引擎 |
| 知识管理 | RAG | 规则文档化后更新方便，不用改代码 |
| 评估架构 | 多Agent拆分 | 每个Agent只负责一个维度，职责清晰、容易调试 |
| 执行方式 | 关键路径并行 | 职业和健康评估互不依赖，并行可减少等待时间 |

### 问题 1：知识库检索总是返回空结果 ⭐⭐⭐⭐⭐

**场景**：工作流搭好后第一次测试，KB 知识检索节点返回空，下游 Agent 拿不到规则，输出全是编造的。

**排查过程**：
1. 先确认知识库文档已上传并索引完成 → 正常
2. 在知识库页面直接用用户输入搜索 → 能召回片段，说明文档本身没问题
3. 检查 KB 检索节点的查询变量配置 → 发现用的是 `{{#start.query#}}`，即用户的原始自然语言输入
4. 分析根因：用户输入是"我30岁，外卖骑手，有高血压，想买重疾险50万"这样的自然语言，而知识库文档里的内容是"3级风险职业 — 加费10%""高血压1级加费20%"这样的结构化规则文本。自然语言和规则文本的语义空间差异太大，Embedding 相似度不够高，导致混合检索也召不回相关片段

**解决方案**：
- 新增了一个 Code 节点「查询构造器」，放在 LLM 信息提取之后、KB 检索之前
- 先从用户输入中提取结构化字段（职业、险种、健康状况、年龄等）
- 再用这些字段拼接成**精准关键词查询**，例如：
  ```
  职业：外卖骑手，职业风险等级与费率调整；
  险种：重疾险，核保规则与要点；
  健康状况：高血压，核保规则与加费标准；
  年龄30岁对应的风险系数
  ```
- KB 检索节点改用这个构造后的查询，而不是原始输入

**效果**：检索召回率从几乎为 0 提升到稳定召回相关规则片段。

**能力体现**：问题定位能力（逐层排查而非盲目改配置）、对 RAG 原理的理解（Embedding 匹配的本质是语义相似度）、工程化思维（在 LLM 和检索之间加一个结构化中间层）。

**面试话术**：
> "我在搭建核保工作流时，遇到一个关键问题：知识库检索一直返回空。我没有盲目改参数，而是逐层排查——先确认文档索引状态、再验证知识库本身的检索能力、最后定位到查询变量的配置。根因是直接用用户的自然语言做 Embedding 查询，但口语化输入和结构化规则文本的语义距离太远。我的方案是在 LLM 提取结构化信息后，用代码节点把关键字段拼接成精准关键词查询再送入检索，这本质上是在用户自然语言和知识库之间加了一层'语义翻译'。这个思路来自于我对 RAG 原理的理解——Embedding 匹配的是语义相似度，不是关键词匹配。"

### 问题 2：LLM 输出格式不稳定，下游 Code 节点频繁报错 ⭐⭐⭐⭐

**场景**：Code 节点的 `json.loads()` 解析 LLM 输出时频繁报错。查看日志发现 LLM 有时输出带 `<think>...</think>` 标签，有时 JSON 前后有多余文字，有时直接输出非 JSON 格式。

**排查过程**：
1. 查看 Code 节点运行日志 → JSON 解析失败的原始输入包含 `<think>` 标签
2. 分析原因：使用的 deepseek-v4-pro 模型在推理时会输出思考过程（`<think>` 标签），即使开了结构化输出也可能出现
3. 另外，随着上下文变长，LLM 偶尔会"不听话"，在 JSON 前后加解释文字

**解决方案**：
- 在所有 Code 节点中实现统一的 `safe_parse()` 函数：
  ```python
  def safe_parse(raw):
      # 1. 用正则清洗 <think>...</think> 标签
      raw = re.sub(r"<think>.*?</think>", "", raw, flags=re.DOTALL).strip()
      # 2. 提取第一个完整的 JSON 对象（找 { 到 } 的范围）
      start = raw.find("{")
      end = raw.rfind("}") + 1
      if start >= 0 and end > start:
          return json.loads(raw[start:end])
      return {}
  ```
- 同时在 LLM 节点配置中开启结构化输出 + 重试机制（最大 3 次，间隔 1000ms）
- 在 Assistant Prefill 中预填 `{` 开头，引导 LLM 输出 JSON

**效果**：系统鲁棒性大幅提升，即使上游 LLM 输出异常，Code 节点也不会崩溃，能降级处理。

**能力体现**：防御性编程思维、对 LLM 输出不确定性有深刻认知、工程化的容错设计。

**面试话术**：
> "LLM 本质上是一个概率系统，输出格式不稳定是常态。我在项目中遇到了 LLM 输出带 think 标签导致下游解析崩溃的问题。我的处理思路是'不信任 LLM 的输出格式，但信任内容的可提取性'——在所有 Code 节点加入安全的 JSON 提取逻辑，清洗标签、容错解析、降级兜底。这体现了我对 AI 产品的一个核心认知：不能把 LLM 当确定性系统来设计，要在工程层面做好容错。"

### 问题 3：Agent 看不到知识库规则，评估结果完全基于模型自身知识 ⭐⭐⭐⭐

**场景**：Agent2（职业风险评估）和 Agent3（健康风险评估）的输出看起来"合理但不精确"，有时给出的加费比例和知识库中的规则不一致。

**排查过程**：
1. 对比 Agent 输出和知识库规则 → 发现有不一致
2. 检查 Agent 的 System Prompt → 写了"从知识库检索结果中匹配规则"
3. 检查 Agent 节点的上下文配置 → **发现未配置**，`{{#context#}}` 获取不到任何内容
4. 根因：Dify 的 LLM 节点不会自动获取上游 KB 节点的检索结果，必须手动在"上下文"配置中指定来源节点

**解决方案**：
- 在 Agent2 和 Agent3 的节点配置中，明确设置"上下文来源"为 `KB:规则检索` 节点
- 在 User Prompt 中使用 `{{#context#}}` 变量引用检索结果
- 将此问题记录为实操手册中的"最易出错的 3 个配置"之一

**效果**：Agent 评估准确度明显提升，结论与知识库规则一致。

**能力体现**：对 Dify 平台机制的深入理解、排查问题的系统性思维、文档沉淀意识。

**面试话术**：
> "做 AI 产品有个容易被忽视的点：配置层面的细节往往比 Prompt 更重要。我的 Agent 评估结果不准，第一反应不是去改 Prompt，而是先检查上下文是否正确传递。果然发现是 Dify 的上下文配置漏了——LLM 节点不会自动获取上游检索结果，必须显式指定。这个经历让我意识到，AI PM 不能只懂 Prompt Engineering，还要理解平台机制和数据流。"

### 问题 4：多个评估维度串行执行，响应延迟过长 ⭐⭐⭐

**场景**：最初的流程是 Agent2（职业）→ Agent3（健康）→ Code（定价）→ Agent4（决策），全部串行。用户输入后要等很久才能看到结果。

**分析**：
- 职业风险评估和健康风险评估是**独立的**——它们评估不同的维度，互不依赖
- 但原有的串行结构把它们变成了顺序执行
- 4 个节点串行约需要 12-16 秒（每个 LLM 调用 3-4 秒）

**解决方案**：
- 重新设计工作流拓扑：Agent2 和 Agent3 从 KB:规则检索 节点并行分支
- Dify 自动并行执行这两个分支，然后汇入 Code:定价计算
- 优化后的延迟约 6-8 秒（并行分支取最慢的那个 + 后续串行节点）

**效果**：延迟减少约 50%。

**能力体现**：系统架构思维、对工作流引擎的理解、性能优化的主动意识。

**面试话术**：
> "这其实是一个简单的架构优化——识别出工作流中的并行机会。职业评估和健康评估是独立的，不应该互相等待。我把工作流从纯串行改为并行分支汇聚，延迟直接减半。这体现的是系统设计思维：不要默认串行，要主动分析节点间的依赖关系，找出可以并行的路径。"

---

### 问题 5：单 LLM 做所有事，准确率上不去 ⭐⭐⭐⭐⭐

**场景**：最初的设计是一个 LLM 同时做信息提取 + 职业评估 + 健康评估 + 决策，Prompt 非常长，但输出质量不稳定——有时职业评估详细但健康评估草率，有时决策逻辑和评估结果矛盾。

**分析**：
- 一个 LLM 处理太多维度时，注意力分散，容易顾此失彼
- 不同评估任务需要不同的知识背景和判断标准，混在一起容易出错
- 长 Prompt 导致 LLM 对指令的遵循度下降（"迷失在中间"问题）

**解决方案**：
- 拆分为 5 个专业化 Agent：
  - Agent1：只做信息提取（从自然语言提取结构化字段）
  - Agent2：只做职业风险评估（匹配职业风险等级和加费）
  - Agent3：只做健康风险评估（匹配健康核保规则和加费）
  - Agent4：只做决策汇总（综合三个维度做最终决定）
  - Agent5：只做报告生成（将结构化结果转为用户可读的 Markdown 报告）
- 每个 Agent 的 System Prompt 精简到只描述一个职责
- 并行评估的 Agent2 和 Agent3 独立配置上下文和输入变量

**效果**：
- 每个维度的评估准确率提高
- 更容易调试（出问题时能快速定位到具体 Agent）
- 方便后续迭代（改职业规则不影响健康评估，改报告格式不影响决策逻辑）

**能力体现**：AI 产品架构设计能力、对 LLM 能力边界的理解、工程化拆解思维。

**面试话术**：
> "这个项目让我最深刻的产品认知是：不要把 LLM 当万能工具箱。我的第一版是一个大 Prompt 让 LLM 做所有事，结果每个维度都做不精。后来我借鉴了 Multi-Agent 架构思想，把核保拆成 5 个专职 Agent——信息提取、职业评估、健康评估、决策汇总、报告生成。每个 Agent 只做一件事，Prompt 短了，准确率反而高了。这跟管理团队一样——一个人什么都干，不如每个人专注一个领域。"

### 问题 6：规则变更需要改代码发版，不符合业务需求 ⭐⭐⭐

**场景**：保险核保规则会频繁调整（监管要求、市场策略），如果规则写在代码里，每次调整都要走开发排期 → 测试 → 发版流程。

**分析**：
- 业务人员希望"改规则像改文档一样简单"
- 代码中的规则（如费率表、风险等级映射）和知识库中的规则需要保持同步
- 但纯 RAG 方案在数值计算上可能不够精确（LLM 可能算错）

**解决方案**：
- 采用"知识库 + 代码双轨"架构：
  - 知识库存储所有文本规则（职业风险等级描述、健康加费标准、年龄系数说明等），供 LLM Agent 推理参考
  - Code 节点存储精确的数值计算逻辑（费率表、年龄系数函数、风险系数公式），保证计算准确性
  - 知识库支持热更新，代码层处理确定性的数值计算
- 规则变更时：文本规则更新知识库文档即可；费率表等核心数值变更需要更新代码（确保计算准确性）

**效果**：大部分规则调整无需发版，直接更新知识库文档。核心费率变更虽然需要代码更新，但有明确的双轨边界。

**能力体现**：产品架构设计能力、对 AI 能力边界的务实判断（不迷信 RAG，也不迷信代码）、业务理解。

**面试话术**：
> "做企业级 AI 产品，有一个关键判断：什么放知识库、什么放代码。我把规则分为两类——需要 LLM 理解和解释的文本规则放知识库，支持热更新；需要精确计算的数值规则放代码，保证准确性。这不是技术选型，是产品决策——我选择牺牲一部分灵活性来换取计算准确性，因为保费算错一块钱都是生产事故。"

---

### 问题 7：用户输入信息不全时系统崩溃 ⭐⭐

**场景**：测试"我想买个保险，不太懂，帮我看看"这种极端输入时，早期版本会崩溃或输出乱码。

**分析**：
- 信息提取 Agent 返回全是"未知"
- 查询构造器只能生成"年龄风险系数表"这一个查询片段
- 定价计算中年龄默认 30、性别默认男，综合系数为 1.0
- 但早期版本没有处理"未知"的默认值逻辑，导致代码报错

**解决方案**：
- 在所有 Code 节点加入完整的默认值兜底逻辑
- 信息提取 Agent 的 Prompt 明确规定：未提及的字段填"未知"
- 查询构造器：当所有字段都是"未知"时，至少查询"年龄风险系数表"
- 定价计算：年龄未知→默认 30、性别未知→默认男性费率、职业未知→1 级风险、健康未知→标准体
- Agent4 决策汇总：当信息不足时输出"信息不足，无法做出准确核保决策，请补充以下信息..."

**效果**：极端输入也不会崩溃，系统优雅降级并引导用户补充信息。

**能力体现**：边界情况处理意识、用户体验思维、防御性设计。

## 三、API 级联排查故事（2026-06-24 新增）

### 4.1 问题背景

完成三个独立模块后，我设计了三模块联动方案——Scheme A（Hub统一路由）和 Scheme B（HTTP API级联编排）。Scheme A 在 Dify Cloud 上顺利跑通，但 Scheme B 导入后一直转圈，所有节点都不执行。

### 4.2 面试话术（"你项目中遇到最难的技术问题是什么？"）

> "最难的其实不是 Prompt 设计或架构选型，而是一个平台兼容性问题。"
>
> "我做 InsureFlow 时，三个模块各自都能独立运行，但我想把它们串联成一条完整链路——用户一次输入，自动走完产品配置→核保→保单服务。我设计了方案——用 Dify 的 HTTP Request 节点调用三个模块的 Workflow API，导入 YML 后发布，结果一直转圈，所有节点都不执行。"
>
> "最头疼的是 Dify 作为低代码平台没有暴露执行日志，传统的'看日志定位修复'路径走不通。我用了控制变量法：先创建 3 节点最小版本（Start→LLM→End），跑通了，说明不是 Dify 基础结构问题。然后增量添加 HTTP 节点，6 节点版本一运行就转圈，锁定了 HTTP Request 节点是问题源。最后我创建了 Debug V3——把 URL 从环境变量拼接改为硬编码，排除变量解析的嫌疑。结果还是转圈，最终确认根因是 Dify Cloud 沙箱环境禁止 HTTP 节点发起出站请求。"
>
> "定位到根因后，我没有纠结于平台的限制，而是写了一个 Python 编排脚本——用标准库直接调用三个模块的 Workflow API，实现完整的三模块 API 级联。这本质上是一个轻量级的 API Gateway + Service Orchestrator。"
>
> "这段经历让我深刻理解到：做 AI 产品，平台本身的限制和兼容性是一等公民问题，和 Prompt 设计、架构选型同等重要。而且，知道什么时候该跳出低代码平台自己造工具，是 AI 产品经理很重要的判断力。"

### 4.3 关键能力展示

| 能力 | 体现 |
|------|------|
| 问题定位 | 无日志环境下用控制变量法三级排查 |
| 平台认知 | 理解低代码平台的能力边界 |
| 技术选型 | 不固执于一种方案，根据约束灵活切换 |
| 工程化思维 | 最小化测试→增量添加→变量隔离的系统排查方法论 |

## 四、面试叙事线（推荐讲述顺序）

### 完整版（3-5 分钟）

```
1. 【背景】我对保险科技感兴趣，通过自学了解了保险业务流程。
   发现核保环节适合用AI辅助——规则明确、流程清晰。

2. 【方案】我用 Dify 搭了一个智能核保助手。
```

## 五、常见追问及回答

### Q1：为什么选 Dify 而不是自己写代码或用 Coze/Dify 以外的平台？

> "我评估过三个方案：纯代码开发灵活性最高但周期太长，不适合快速验证；
> Coze 在对话型应用上体验更好，但工作流编排能力和知识库管理不如 Dify；
> Dify 的可视化工作流 + 知识库 + 多模型支持正好匹配我的需求——
> 多 Agent 并行、RAG 检索、Code 节点做计算，这些在 Dify 里都能实现。
> 而且 Dify 开源的，部署在公司内网也不会有数据安全问题。"

### Q2：核保的准确性怎么保证？怎么知道 Agent 评估得对不对？

> "我做了三层验证：第一层是测试用例——设计 8 个覆盖全场景的用例，
> 每个用例有预期的风险系数和保费，对比系统实际输出；
> 第二层是 Code 节点做数值计算的硬约束——费率表和计算公式是确定的，
> LLM 只负责规则匹配和文字推理，不碰数字；
> 第三层是每个 Agent 的 System Prompt 明确要求'不使用知识库之外的规则'，
> 加上 Temperature 设到 0.3 降低随机性。"

### Q3：这个项目如果继续做，你会怎么迭代？

> "三个方向：第一，引入用户反馈机制，让核保员可以标记'同意/不同意'每一条 Agent 判断，
> 积累标注数据后做 Prompt 优化甚至微调；第二，接入真实的保险公司核心系统 API，
> 让 Demo 变成可实际调用的原型；第三，增加 A/B 测试框架，
> 对比不同 Prompt 策略和模型选择对核保结论准确率的影响。"

### Q4：你觉得这个项目最大的不足是什么？

> "知识库的规则文档是我手写的，质量和覆盖度有限。
> 真实场景中需要对接保险公司的核保手册、监管文件、历史核保案例，
> 而且规则的冲突处理（比如两条规则给出不同加费比例）目前没有很好的机制。
> 另外，缺少量化的准确率评估——比如找 100 个真实核保案例，对比 AI 决策和人工决策的一致性。"
