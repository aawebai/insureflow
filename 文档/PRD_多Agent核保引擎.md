## 一、产品概述

### 1.1 产品定位

面向保险公司的**AI原生核保决策引擎**。用户用自然语言描述个人信息（年龄、职业、健康状况），系统通过5个专业化Agent协同评估，结合RAG知识库输出结构化核保决策报告和精准定价。

**一句话描述**：把保险核保从"查规则手册+人工判断"变成"自然语言输入→多Agent并行评估→一键出结论"。

### 1.2 目标用户

| 用户角色 | 场景 | 核心诉求 |
|----------|------|----------|
| **保险代理人/经纪人** | 客户面前实时核保 | 快速出结论、解释推理过程、辅助销售 |
| **核保员** | 处理复杂案件 | 标准化评估框架、规则自动匹配、降低遗漏 |
| **C端消费者** | 投保前自查 | 了解自己能否投保、大概保费多少、哪些因素影响 |
| **产品经理（内部）** | 调整核保规则 | 热更新规则文档、即时生效、不用排期发版 |

### 1.3 解决的痛点

| # | 痛点 | 传统做法 | AI改进 |
|---|------|----------|--------|
| 1 | **核保门槛高** | 核保员需熟记数百条规则，培训周期3-6个月 | 自然语言输入，Agent自动匹配规则，解释推理过程 |
| 2 | **规则更新慢** | 规则硬编码在代码中，改一条规则需排期2-4周 | RAG知识库驱动，更新文档即时生效 |
| 3 | **决策不透明** | 结论是黑盒，客户问"为什么加费"核保员不一定能解释清楚 | Agent输出推理链，每条结论附带规则依据 |
| 4 | **多维度评估串行低效** | 职业、健康、年龄等维度逐一评估，耗时长 | 独立维度并行评估，延迟减半 |
| 5 | **一致性难保证** | 不同核保员对同一案件的判断可能不同 | 统一规则源 + 标准化评估流程 |

## 二、功能范围

### 2.1 MVP范围（已实现）

| 功能 | 描述 | 优先级 | 状态 |
|------|------|:---:|:---:|
| **F1: 自然语言信息提取** | 从用户口语化输入中提取年龄/职业/健康状况/险种/保额等结构化字段 | P0 | ✅ 已实现 |
| **F2: 结构化查询构造** | 将提取字段拼接为精准知识库查询，解决自然语言与规则文本的语义鸿沟 | P0 | ✅ 已实现 |
| **F3: RAG规则检索** | 从核保规则知识库检索相关规则片段（Top K=8，混合检索+重排序） | P0 | ✅ 已实现 |
| **F4: 职业风险评估** | 独立Agent评估职业风险等级（1-6级），匹配加费比例 | P0 | ✅ 已实现 |
| **F5: 健康风险评估** | 独立Agent评估健康状况（高血压/糖尿病/BMI/吸烟等9类），匹配加费标准 | P0 | ✅ 已实现 |
| **F6: 年龄+性别+吸烟风险评估** | Code节点精确计算年龄系数、性别系数、吸烟系数 | P0 | ✅ 已实现 |
| **F7: 多维风险融合定价** | 安全解析3个Agent输出，查费率表，计算综合风险系数和保费 | P0 | ✅ 已实现 |
| **F8: 四档决策输出** | 标准体/次标准体/延期体/拒保体，每档含决策理由 | P0 | ✅ 已实现 |
| **F9: Markdown报告生成** | 结构化核保报告，含决策结论/风险明细/保费计算/建议 | P0 | ✅ 已实现 |
| **F10: 降级规则兜底** | KB检索失败时，System Prompt内嵌核心规则作为兜底 | P0 | ✅ 已实现 |
| **F11: LLM输出容错** | safe_parse清洗think标签+JSON提取+默认值兜底 | P0 | ✅ 已实现 |

## 三、用户流程

### 3.1 主流程

```
用户输入自然语言信息
      │
      ▼
┌─────────────────────────────────────────────────────┐
│  "我30岁，是一名外卖骑手，有轻度高血压，平时抽烟，  │
│   想买50万保额的重疾险，帮我看看能投保吗？"          │
└─────────────────────────────────────────────────────┘
      │
      ▼
  [Agent1: 信息提取]  ← 提取结构化JSON
      年龄:30, 职业:外卖骑手, 健康状况:[高血压(轻度)]
      险种:重疾险, 保额:50万, 吸烟:是
      │
      ▼
  [查询构造器]  ← 拼接精准查询关键词
      "职业：外卖骑手，职业风险等级与加费比例；
       险种：重疾险，基准费率；
       健康状况：高血压(轻度)，加费标准；
       吸烟：是，加费标准"
      │
      ▼
  [知识库检索]  ← 从核保规则库检索相关规则片段
      │
      ├─────────────────────────────────────┐
      ▼                                     ▼
  [Agent2: 职业风险评估]              [Agent3: 健康风险评估]
   → 外卖骑手=3级风险职业             → 轻度高血压=加费15-25%
   → 加费10%                          → 吸烟=加费10-50%
      │                                     │
      └──────────────┬──────────────────────┘
                     ▼
  [定价计算]  ← Code节点精确计算
      基准保费: 11,000/年
      职业系数: 1.10
      健康系数: 1.20
      年龄系数: 1.00
      吸烟系数: 1.15
      综合系数: 1.10×1.20×1.00×1.15 = 1.518
      最终保费: 11,000×1.518 = 16,698/年
      │
      ▼
  [Agent4: 决策汇总]
      → 次标准体承保（加费约52%）
      → 决策理由: 职业3级风险+高血压+吸烟
      │
      ▼
  [Agent5: 报告生成]
      → 输出Markdown格式核保报告
```

### 3.2 边缘场景处理

| 场景 | 用户输入示例 | 系统行为 |
|------|-------------|----------|
| 信息极少 | "想买保险，不太懂" | 所有字段默认值兜底，输出"信息不足"提示并引导用户补充 |
| 信息矛盾 | "我20岁，已退休" | Agent提取为"20岁+退休"，查询构造器仍正常构造，定价时年龄系数按实际年龄计算 |
| 敏感职业 | "我是雇佣兵" | 职业6级 → 拒保体，Agent4输出拒保决策并解释原因 |
| 多疾病叠加 | "高血压+糖尿病+BMI 32" | Agent3分别评估3个因子，定价时取最大加费比例+额外叠加系数 |
| LLM输出异常 | 输出带`<think>`标签 | safe_parse清洗 → 提取JSON → 降级默认值，系统不崩溃 |
| KB检索为空 | KB返回0条结果 | System Prompt内嵌降级规则兜底，Agent仍可基于内置规则做评估 |

## 四、功能需求明细

### F1: 自然语言信息提取

| 属性 | 内容 |
|------|------|
| 输入 | `INSURE_inquiry` (text, max 256字) |
| 输出 | JSON: `{age, gender, occupation, health_conditions, product_type, sum_insured, smoking_status, bmi}` |
| LLM温度 | 0.7 |
| 结构化输出 | 开启 |
| 重试 | 最大3次，间隔1000ms |
| 异常处理 | 解析失败→所有字段默认null→下游Code节点兜底 |

**Prompt核心设计**：
- 明确字段定义（如health_conditions是数组）
- 未提及字段填null（不编造）
- 支持模糊表达解析（"轻度高血压"→severity: mild, condition: hypertension）

### F2: 结构化查询构造

| 属性 | 内容 |
|------|------|
| 节点类型 | Code（Python） |
| 输入 | LLM1的JSON输出 |
| 输出 | `search_query` (拼接关键词) + `clean_user_data` (清洗后的用户数据) |
| 核心逻辑 | 1. safe_parse清洗think标签 2. 遍历非null字段拼查询 3. 每一行用分号分隔 |

### F3: RAG规则检索

| 属性 | 内容 |
|------|------|
| 检索来源 | 核保规则知识库（8类规则文档） |
| Top K | 8 |
| 检索方式 | 混合检索（语义+关键词） |
| 重排序 | qwen3-rerank |
| 查询变量 | `{{#代码执行:查询构造器.search_query#}}`（非原始用户输入） |

### F4-F5: 并行风险评估

| 属性 | Agent2（职业风险） | Agent3（健康风险） |
|------|-------------------|-------------------|
| 上下文来源 | KB检索节点 | KB检索节点 |
| System Prompt | 含职业6级分类+加费降级规则 | 含9类健康因子+加费降级规则 |
| 输出格式 | JSON: `{risk_level, risk_name, surcharge_ratio, reason}` | JSON: `{health_factors[], overall_surcharge, reason}` |
| 温度 | 0.7 | 0.7 |

### F6: 定价计算

| 属性 | 内容 |
|------|------|
| 节点类型 | Code（Python） |
| 核心算法 | `基准保费 × 职业系数 × 健康系数 × 年龄系数 × 吸烟系数` |
| 费率表 | 内嵌在代码中（重疾/医疗/定寿/意外4套基准） |
| 安全解析 | safe_parse × 3次（职业输出+健康输出+用户数据），每次独立容错 |
| 输出 | `{base_premium, occupation_coeff, health_coeff, age_coeff, smoking_coeff, combined_coeff, final_premium, pricing_detail}` |

**费率表结构示例**：
```
重疾险基准保费（30岁/男/50万保额）：11,000元/年
年龄系数：≤17(0.8) 18-25(0.9) 26-35(1.0) 36-45(1.3) 46-55(1.6) 56-65(2.0) >65(拒保)
性别系数：男(1.0) 女(0.95)
```

### F7: 决策汇总

| 属性 | 内容 |
|------|------|
| 输入 | 定价计算的完整JSON |
| 输出 | 四档决策之一 + 综合风险评分 + 决策理由 |

**决策标准**：
| 决策档位 | 触发条件 | 说明 |
|----------|----------|------|
| 标准体 | 综合系数 ≤ 1.15 | 正常费率承保 |
| 次标准体 | 综合系数 1.15-2.5 | 加费承保 |
| 延期体 | 存在可逆风险因子 | 建议康复后再投保 |
| 拒保体 | 综合系数 > 2.5 或命中拒保规则 | 拒绝承保，推荐替代险种 |

### F8: 报告生成

Markdown格式，固定包含5部分：
1. **核保结论**（决策档位+一句话总结）
2. **风险因子明细**（职业/健康/年龄/吸烟，逐项打分）
3. **保费计算**（基准保费 × 各系数 = 最终保费）
4. **决策依据**（引用的核保规则来源）
5. **后续建议**（加费体→解释原因；延期体→建议复查时间；拒保体→替代险种推荐）

## 八、风险与假设

| # | 风险 | 影响 | 缓解措施 |
|---|------|------|----------|
| R1 | 知识库规则不完整导致Agent误判 | 高 | System Prompt内嵌降级规则 + 定期审核KB覆盖度 |
| R2 | LLM输出波动导致定价计算错误 | 高 | Code节点做精确计算，LLM不碰数字 |
| R3 | 并行评估结果互相矛盾 | 中 | Agent4决策汇总时做矛盾检测，标记需人工复核 |
| R4 | 知识库文档格式变更影响检索 | 中 | 制定KB文档格式规范，变更后回归测试 |
| R5 | 模型切换导致Prompt不兼容 | 低 | 保留模型切换回归测试用例集 |

### 关键假设

- A1: 核保规则可以用文本形式充分表达（对规则复杂度过高的险种可能不成立）
- A2: 用户能够用自然语言准确描述自身健康情况（对隐瞒病史的情况无法识别）
- A3: 保险监管允许使用AI辅助核保决策（需持续关注合规要求）
- A4: DeepSeek模型持续可用且不出现重大性能退化

### B. 术语表

| 术语 | 说明 |
|------|------|
| Agent | 专业化LLM节点，每个Agent只负责一个维度的评估 |
| RAG | Retrieval-Augmented Generation，检索增强生成 |
| 标准体 | 健康/职业/年龄均在正常范围，按标准费率承保 |
| 次标准体 | 存在一定风险因子，需额外加费承保 |
| 延期体 | 当前风险不可接受但未来可能改善，建议延期投保 |
| 拒保体 | 风险过高，拒绝承保 |
| safe_parse | 防御性JSON解析函数，清洗think标签+提取JSON+默认值兜底 |
| KB | Knowledge Base，知识库 |
