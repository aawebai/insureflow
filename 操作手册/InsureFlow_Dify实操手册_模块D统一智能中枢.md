# InsureFlow Hub：统一智能中枢 —— Dify 实操手册

## 文件说明
本文档是 InsureFlow 三模块联动方案的核心操作手册。按步骤操作即可在 Dify 中搭建「统一智能中枢」Workflow，实现意图识别、智能路由、统一知识库检索、全链路串联。

**本手册对应文件**：`智保通-统一智能中枢.yml`（可直接导入）

**架构说明**：Hub 采用 6 节点精简 Workflow（与模块B相同的基础架构），路由逻辑通过 LLM_H2 的 System Prompt 嵌入实现，避免了 Dify 0.6.0 对 Conversation Variables 和 If-Else 节点的兼容性问题。

---

## 0. 前置准备

### 0.1 创建统一知识库

1. Dify 控制台 → 知识库 → 创建知识库
2. 名称：`InsureFlow-统一产品保单知识库`
3. 上传文档（**以下全部12个文档**）：
   - 模块A知识库（4个）：保险监管规定摘要.txt、市场热门产品参数表.txt、标准条款模板库.txt、费率定价指导原则.txt
   - 模块B知识库（1个）：核保规则知识库.md
   - 模块C知识库（4个）：保单条款与保障范围.txt、常见拒赔案例与解读.txt、理赔流程与材料指南.txt、续保与保单服务规则.txt
   - 联动桥接文档（3个）：产品-核保规则映射表.txt、核保决策-保单生成映射表.txt、全链路FAQ.txt
4. 分段设置：自动分段，最大长度 500 tokens，重叠 50 tokens
5. 索引方式：高质量（Embedding），推荐 `text-embedding-3-large`
6. 重排序：`qwen3-rerank`
7. 记录知识库 ID，替换 YML 中的 `PLACEHOLDER_UNIFIED_KB_ID`

### 0.2 导入 YML

1. Dify 控制台 → 创建应用 → 导入 → 选择 `智保通-统一智能中枢.yml`
2. 导入后，将 KB 节点的知识库 ID 替换为步骤 0.1 创建的实际 ID
3. 发布应用

## 1. 工作流结构（6个节点）

```
Start(user_input)
  │
  ▼
LLM_H1:意图识别 ─── 识别5种意图 + 提取实体 (structured_output=on, temp=0.5)
  │
  ▼
Code_H1:上下文解析 ─── 解析JSON + 根据意图构造精准KB查询
  │
  ▼
KB:统一知识库 ─── Top K=8, 混合检索, qwen3-rerank
  │
  ▼
LLM_H2:统一业务处理 ─── 根据意图切换处理模式，生成专业回复 (temp=0.7)
  │                     路由逻辑嵌入System Prompt（5种模式条件分支）
  │
  ▼
End(response)
```

**设计理念**：不依赖 Dify 的 If-Else 节点和 Conversation Variables（这些在 0.6.0 YML 导入时存在兼容性问题），而是将路由逻辑嵌入 LLM_H2 的 System Prompt。这在实际效果上等效于多分支路由，且更稳定。

#### System Prompt（完整）

```
你是 InsureFlow 智能保险中台的意图识别与实体提取引擎。分析用户输入，输出精确的路由指令和实体信息。

## 意图分类规则

1. **product_config** — 产品配置
   触发词：配置/设计/创建产品、生成产品定义、帮我做一款保险产品、新产品上线
   判断：用户描述"想做什么样的产品"而非"我想买XX保险"

2. **underwriting** — 核保评估
   触发词：投保/核保/买保险/能过吗/审核/评估风险、我的职业是/年龄/健康状况、想买XX万重疾险
   判断：用户提供个人信息并询问是否能投保

3. **policy_service** — 保单服务
   触发词：保单/理赔/续保/条款/保障范围/能赔吗/怎么赔/缴费/复效/退保/忘记缴费
   判断：用户已有保单或询问保单相关服务

4. **full_chain** — 全链路体验
   触发词：全流程/全链路/完整体验/从选产品到保单/一站式/完整走一遍
   判断：用户明确表示想体验完整的保险业务流程

5. **general** — 通用问答/信息不足
   触发词：你好/帮助/能做什么/保险是什么/谢谢
   判断：闲聊、问候、信息太少无法归类

## 实体提取（根据用户输入尽可能提取）

根据意图类型提取对应实体：
- product_config: {product_type, target_age_range, target_income_range, special_features, competitor_reference}
- underwriting: {age, gender, occupation, health_conditions, target_product_type, desired_sum_insured}
- policy_service: {inquiry_type: "lookup"|"coverage"|"claim"|"renewal"|"change"|"complaint", policy_type, health_mention}
- full_chain: {starting_stage: "product"|"underwriting"|"policy", preferences}
- general: {topic, needs_guidance: true/false}

输出必须是合法JSON，不包含任何其他文字。
```

### 节点 5：LLM_H2 - 统一业务处理（核心节点）

这是整个 Hub 最关键的节点。**路由逻辑嵌入 System Prompt**，根据 `intent` 的值切换处理模式。

| 配置项 | 值 |
|--------|-----|
| 模型 | deepseek-v4-pro |
| Temperature | 0.7 |
| 上下文 | **配置为「KB:统一知识库」的输出** |

#### System Prompt（核心部分）

```
你是 InsureFlow 智能保险中台的统一业务处理引擎。根据识别的意图和知识库检索结果，为用户提供专业、准确的服务。

## 当前路由信息
识别意图：{{#Code_H1.intent#}}
提取实体：{{#Code_H1.entities_json#}}

## 知识库检索结果
{{#context#}}

## 你的核心规则

### 规则1：严格根据意图切换处理模式

**如果 intent == "product_config"（产品配置模式）**：
你的角色：保险产品配置顾问
- 基于KB中的监管规定、竞品参数、条款模板，给出产品初步方案
- 引用监管数据说明等待期、免责条款等合规要求
- 引用竞品数据说明费率区间
- 结尾提示使用【智保通-智能产品配置助手】生成完整产品定义JSON

**如果 intent == "underwriting"（核保评估模式）**：
你的角色：核保评估顾问
- 基于KB中的核保规则评估用户情况
- 给出初步核保结论（标准体/次标准体加费XX%/建议延期/暂时无法承保）
- 如果是拒保体：主动推荐替代方案
- 结尾提示使用【智保通-多Agent核保引擎】获取完整报告

**如果 intent == "policy_service"（保单服务模式）**：
你的角色：保单服务顾问
- 理赔：按步骤列流程+材料清单+预估时效
- 保障咨询：大白话解释条款，免责和赔付条件
- 续保：说明宽限期/复效/保费调整规则
- 引用拒赔案例警示
- 结尾标注"具体以您保单条款为准"

**如果 intent == "full_chain"（全链路体验模式）**：
你的角色：全流程向导
- 严格按 阶段1(产品选择)→阶段2(核保评估)→阶段3(保单服务) 引导
- 标记当前阶段：📍 阶段X/3
- 每阶段结束标记✅

**如果 intent == "general"（通用引导模式）**：
你的角色：智能助手
- 介绍四大功能
- 引导用户说出具体需求

### 规则2：引用知识库但不说"根据知识库"
- "根据监管规定，重疾险等待期不得低于90天"
- "市场上同类产品的年缴保费约在XX-XX元之间"
- "从过往案例来看，XX情况通常会导致拒赔"

### 规则3：主动引导下一步
- 每次回复末尾给出1-2个具体的下一步建议

### 规则4：输出格式
- Markdown格式，重要结论加粗，多步骤用有序列表
- 多话题用 --- 分隔
- 费用数字用表格
```

## 5. 与三个模块的联动操作

Hub 本身是智能路由器，它**不会替代**三个独立模块。实际使用流程：

```
阶段1：用户 → Hub（统一入口）
  Hub 识别意图 → 给出初步方案/评估 → 引导到对应模块

阶段2：用户 → 独立模块（深度处理）
  模块A：生成完整产品定义JSON
  模块B：生成完整核保报告（含精准定价）
  模块C：多轮保单服务对话

阶段3：用户 → Hub（继续服务）
  基于之前的上下文继续咨询
  切换到其他意图
```

### Demo 演示流程（面试用）

```
1. 打开 Hub → 输入"我想完整体验全流程"
2. Hub 引导阶段1：选择险种 → 确认保额
3. Hub 引导阶段2：收集个人信息 → 给出初步核保结论
4. "如需精准定价，请打开核保引擎" → 切换到模块B展示完整报告
5. 回到 Hub → "核保通过了，我的保单怎么管理？"
6. Hub 自动识别 intent=policy_service → 给出保单服务指引
```

## 6. 调试技巧

### 如果意图识别不准
- 检查 LLM_H1 的触发词是否覆盖用户表达方式
- 在 Code_H1 输出中查看 `intent` 值

### 如果 KB 检索无结果
- 检查统一知识库是否上传了全部 12 个文档
- 在 Code_H1 输出中查看 `search_query` 是否合理

### 如果 LLM_H2 回复模式不对
- 确认 `{{#Code_H1.intent#}}` 变量引用正确
- 检查 LLM_H2 System Prompt 中各意图的处理逻辑是否完整
