# InsureFlow 模块C：智能保单管家 —— Dify 实操手册

## 文件说明
本文档对应设计方案 v2.0 中的 **P2 模块：智能保单管家（Policy Advisor）**。  
按步骤操作即可在 Dify Chatflow 中搭建完整可运行的多轮对话保单服务系统。

**与模块A/B的核心差异**：本模块是 **Chatflow**（对话流），不是 Workflow。因为保单服务本质是多轮对话——用户可能先问保单内容，下一轮追问理赔，且每次追问依赖上一轮对话上下文。Chatflow 支持 Conversation Variables（对话变量）跨轮次记忆。

最可能出错的3个配置
  1. Conversation Variables 必须在 Chatflow 顶部面板中声明，否则 LLM 节点的 Prompt 引用会报错
  2. 条件路由的 if-else 节点每条分支必须配置完整的条件表达式，且 default 分支必须存在
  3. 4 个分支的 KB 检索+LLM 处理节点必须全部汇入 LLM_C3:回复整合，否则某些分支走到尽头无输出

---

## 0. 前置准备：创建保单服务知识库

### 步骤 0.1：在 Dify 中创建知识库
1. Dify 控制台 → 顶部导航「知识库」→ 「创建知识库」
2. 知识库名称：`保单服务知识库`
3. 上传文档（见步骤 0.3 的文件清单）：
   - `保单条款与保障范围.txt`
   - `理赔流程与材料指南.txt`
   - `常见拒赔案例与解读.txt`
   - `续保与保单服务规则.txt`
4. 分段设置：
   - 分段方式：自动分段与清洗
   - 分段最大长度：500 tokens
   - 重叠长度：50 tokens
5. 索引方式：高质量（Embedding）
6. 完成后记录知识库 ID，替换 YML 中的占位符

### 步骤 0.2：确认模型
- deepseek-v4-pro（LLM 节点）
- text-embedding-3-large（Embedding）
- qwen3-rerank（重排序，tongyi 供应商）

### 步骤 0.3：知识库文档清单

| 文件 | 核心内容 | 主要服务场景 |
|------|------|:---:|
| 保单条款与保障范围.txt | 四大险种的保障范围、免责条款、等待期、理赔条件、保额规则 | 保单查询 / 保障咨询 |
| 理赔流程与材料指南.txt | 各险种理赔步骤(1-4步)、所需材料清单、处理时效、到账时间 | 理赔问询 |
| 常见拒赔案例与解读.txt | 10个典型拒赔案例场景、拒赔原因、如何避免、用户注意事项 | 保障咨询（预期管理） |
| 续保与保单服务规则.txt | 宽限期规则、复效条件、续保流程、保费调整说明、保单变更 | 续保提醒 |

## 1. 创建 Chatflow

### 步骤 1.1
1. Dify 控制台 → 「创建应用」→ 选择「对话流（Chatflow）」
2. 名称：`智保通-智能保单管家`
3. 描述：`面向已承保客户的AI保单管家——保单解读、保障咨询、理赔指引、续保提醒`
4. 开场白（在应用设置中配置）：
   ```
   您好！我是您的智能保单管家。我可以帮您：
   - 📋 查询保单信息和缴费状态
   - 🔍 了解保障范围（什么能赔、什么不赔）
   - 📝 获取理赔流程和材料清单
   - ⏰ 了解续保规则和注意事项

   请直接告诉我您需要什么帮助～
   ```

### 步骤 1.2：配置 Conversation Variables

在 Chatflow 画布顶部的「对话变量」面板中添加：

| 变量名 | 类型 | 描述 |
|--------|------|------|
| `user_policy_type` | string | 用户投保险种（从对话中提取） |
| `user_health_concern` | string | 用户提到的健康问题（用于后续追问） |
| `user_last_intent` | string | 上一轮对话意图（用于代词消解） |
| `user_coverage_amount` | string | 用户保额（从对话中提取） |
| `conversation_round` | number | 当前对话轮数（用于判断是新用户还是老用户） |

## 2. 节点配置（共 10 个节点）

### Chatflow 全局结构
```
Start → LLM_C1:意图识别 → If-Else:意图路由
                              │
          ┌───────────────────┼───────────────────────┬──────────────────┐
          ▼                   ▼                       ▼                  ▼
    [保单查询]           [保障咨询]               [理赔问询]          [续保/其他]
          │                   │                       │                  │
          ▼                   ▼                       ▼                  ▼
    KB_C2a:              KB_C2b:                  KB_C2c:             KB_C2d:
    保单条款检索          保障范围检索             理赔规则检索         续保规则检索
          │                   │                       │                  │
          ▼                   ▼                       ▼                  ▼
    LLM_C2a:             LLM_C2b:                 LLM_C2c:            LLM_C2d:
    保单解读              保障分析                 理赔指引             续保建议
          │                   │                       │                  │
          └───────────────────┼───────────────────────┴──────────────────┘
                              ▼
                      LLM_C3:回复整合
                              │
                              ▼
                           Answer
```

### 节点 2：LLM_C1 - 意图识别+信息提取

| 配置项 | 值 |
|--------|-----|
| 类型 | LLM |
| 节点名 | `LLM_C1:意图识别` |
| 模型 | deepseek-v4-pro |
| 上下文 | **不配置** |
| 记忆 | **开启**（窗口大小10轮，Chatflow 核心能力） |

#### 提示词配置

**System Prompt**：
```
你是一个保险客服意图识别器。分析用户在保单服务场景下的意图，同时提取关键信息。

意图分类（可同时识别多个）：
1. policy_lookup - 保单查询（"我的保单还有效吗""下次什么时候缴费""保额是多少""什么时候买的"）
2. coverage_inquiry - 保障咨询（"这个病在保障范围吗""能赔多少""什么情况不赔""有没有免赔额"）
3. claim_guidance - 理赔指引（"怎么理赔""需要什么材料""去哪里办理""多久能赔下来""出险了怎么办"）
4. renewal_reminder - 续保相关（"快到期了怎么续""保费涨了吗""忘缴费了怎么办""要不要续"）
5. general_inquiry - 通用咨询（不属于以上四类的保单服务问题）

重要信息提取（只提取用户明确提到的，不编造）：
- health_condition: 用户提到的疾病/症状/健康状况，如"高血压""甲状腺结节"
- treatment_type: 提到的治疗方式，如"住院""手术""门诊""检查"
- date_reference: 时间相关表述，如"上个月""去年""下周"
- policy_reference: 提到的保单信息，如保单号、险种名称
- urgency_hint: 紧急信号，投诉或严重情况的用词

代词消解规则：
- 如果用户使用"那""这个""它""这种情况"等代词，请结合对话历史推断具体指代内容
- 例如：上一轮问了"我的重疾险"，本轮"那如果我得了结节呢" → health_condition应为"结节"，policy_type继承上一轮的"重疾险"
- 如果对话历史为空或无法推断，直接按字面提取

当前对话轮数：第{{#conversation_round#}}轮（0表示新对话）

输出必须是合法JSON，不包含任何其他文字。intent字段为数组（支持多意图），优先级从高到低排列。
```

**User Prompt**：
```
用户本轮输入：{{#sys.query#}}

请识别意图并提取信息。如用户使用代词指代之前提到过的内容，请结合对话历史还原。
```

**Assistant Prefill**：
```json
{
  "intent": ["
```

#### 关键设置
| 设置项 | 值 |
|--------|-----|
| 结构化输出 | **开启** |
| 重试 | 开启，最大3次，间隔500ms |
| Temperature | 0.3（意图识别需要精准分类） |
| 记忆 | **开启**（窗口10轮，这是Chatflow多轮对话的核心） |

### 节点 3：If-Else - 意图路由

| 配置项 | 值 |
|--------|-----|
| 类型 | 条件分支（if-else） |
| 节点名 | `If-Else:意图路由` |
| 输入变量 | `LLM_C1:意图识别` → `text` |

#### 分支条件配置

由于 LLM_C1 输出是 JSON 字符串，条件判断使用 `contains`（包含匹配）：

| 分支 | 条件 | 目标节点 | 说明 |
|------|------|----------|------|
| 分支1 | `text` 包含 `policy_lookup` | KB_C2a:保单条款检索 | 保单查询 |
| 分支2 | `text` 包含 `coverage_inquiry` | KB_C2b:保障范围检索 | 保障咨询 |
| 分支3 | `text` 包含 `claim_guidance` | KB_C2c:理赔规则检索 | 理赔问询 |
| 分支4（default） | 以上均不匹配 | KB_C2d:续保规则检索 | 续保+通用兜底 |

> **注意**：条件按从上到下的顺序匹配。如果用户输入包含多个意图（如 "policy_lookup" 和 "coverage_inquiry" 同时出现），第一个匹配的分支生效。多意图场景由 LLM_C3 在处理过程中通过追问补充。

| 配置项 | 值 |
|--------|-----|
| 类型 | 知识检索（knowledge-retrieval） |
| 节点名 | `KB_C2a:保单条款检索` |

```
- 先直接回答用户的核心问题（是/否/具体数字）
```

### 节点 5b：LLM_C2b - 保障分析

| 配置项 | 值 |
|--------|-----|
| 类型 | LLM |
| 节点名 | `LLM_C2b:保障分析` |
| 模型 | deepseek-v4-pro |
| 上下文 | `KB_C2b:保障范围检索` |
| Temperature | 0.5（保障判断需相对严谨） |

**System Prompt**：
```
你是一个保险保障分析专家。根据知识库中的保障范围、免责条款和理赔条件，帮用户判断特定情况是否在保障范围内。

回复规则：
1. 先给出明确判断：在保障范围 / 不在保障范围 / 需要进一步确认
2. 引用知识库中的具体条款作为判断依据
3. 如果知识库没有足够信息给出确定回答，列出"大概率可以赔的情况"和"大概率不能赔的情况"
4. 如果在保障范围内，说明大概能赔多少（比例/金额）
5. 如果不在保障范围内，解释原因并给出替代建议（如附加险是否覆盖）

判断逻辑：
- 先查保障范围：用户描述的情况是否属于合同约定的保险责任？
- 再查免责条款：是否存在明确的免责情形？
- 再查理赔条件：是否符合理赔的触发条件（如住院天数、疾病定义标准）？
- 最后查赔付规则：能赔多少？有无免赔额？有无给付比例限制？
```

**User Prompt**：
```
用户问题：{{#sys.query#}}

知识库检索到的保障规则与条款：
{{#context#}}

请分析该情况是否在保障范围内。
```

### 节点 6：LLM_C3 - 回复整合

| 配置项 | 值 |
|--------|-----|
| 类型 | LLM |
| 节点名 | `LLM_C3:回复整合` |
| 模型 | deepseek-v4-pro |
| 上下文 | **不配置** |
| Temperature | 0.7 |
| 记忆 | **开启** |

#### 提示词配置

**System Prompt**：
```
你是一个保险客服助手。你的任务是将前置节点的分析结果转化为友好、清晰、专业的用户回复。

回复格式规则：
1. 使用 Markdown 格式，适当使用标题、加粗、列表
2. 关键数字用**加粗**突出
3. 注意事项用 > 引用块标注
4. 重要提醒前面加❗

信息不足时的处理：
- 如果前置节点信息不足以回答用户问题，主动追问具体信息
- 追问时给出示例，引导用户更方便地回答
- 例如："请问您具体是什么情况呢？比如是哪种疾病或症状？做过什么检查吗？"

无关问题处理：
- 如果用户询问跟保单服务无关的话题，礼貌引导回保单服务范围
- 例如："我是您的保单管家，目前可以帮您查询保单、了解保障、获取理赔指引。您有什么保单相关的问题吗？"

结尾规则：
- 每次回复结尾加上："如有其他问题，随时告诉我～"
- 如果涉及具体条款解读，加上免责："以上解读基于通用条款，具体以您的正式保单为准。如需确认，建议拨打客服热线。"
```

**User Prompt**：
```
当前用户意图：{{#LLM_C1:意图识别.text#}}

分支处理结果（根据意图路由，以下之一有值，其余为空）：

【保单查询结果】
{{#LLM_C2a:保单解读.text#}}

【保障分析结果】
{{#LLM_C2b:保障分析.text#}}

【理赔指引结果】
{{#LLM_C2c:理赔指引.text#}}

【续保服务结果】
{{#LLM_C2d:续保与通用服务.text#}}
```

## 4. 关键配置检查清单

| 序号 | 检查项 | 正确的配置 | 常见错误 |
|------|--------|------------|----------|
| 1 | Conversation Variables | 在 Chatflow 顶部面板中声明 5 个变量 | 只在 Prompt 中引用但未声明 |
| 2 | LLM_C1 记忆 | 开启，窗口10轮 | 未开启导致无法处理代词指代 |
| 3 | LLM_C3 记忆 | 开启 | 未开启导致回复缺乏连贯性 |
| 4 | If-Else 默认分支 | default 分支必须配置 | 遗漏 default 导致未匹配意图时无输出 |
| 5 | 4 个 KB 节点知识库 | 均指向「保单服务知识库」 | 选错知识库 |
| 6 | LLM_C2x 上下文 | 各自指向对应的 KB 节点 | 漏配导致 LLM 看不到检索内容 |
| 7 | Answer 回复来源 | 指向 LLM_C3 的 text | 直接指向分支 LLM，跳过了整合 |
| 8 | 开场白 | 在应用设置中配置 | 未配置导致用户打开后不知道能做什么 |

## 5. 测试用例

### 测试 1：保单查询
```
第1轮输入：我买了一份重疾险，帮我看看我的保单还有效吗？

预期：
- intent: [policy_lookup]
- 走保单查询分支
- 回复：告知如何确认保单有效性（缴费状态/宽限期），追问保单号或投保时间
```

### 测试 2：保障咨询
```
输入：我最近查出甲状腺结节，我的重疾险能赔吗？

预期：
- intent: [coverage_inquiry]
- health_condition: 甲状腺结节
- 走保障咨询分支
- 回复：分析甲状腺结节是否在保障范围（首次确诊重疾才赔，单纯结节不属于重疾），说明什么情况下才能触发理赔
```

### 测试 3：理赔指引
```
输入：我上个月住院做了手术，怎么申请理赔？

预期：
- intent: [claim_guidance]
- 走理赔指引分支
- 回复：列出理赔步骤(1报案→2准备材料→3提交→4等结果)，每步标注材料清单和时效
```

### 测试 4：续保
```
输入：我的医疗险快到期了，怎么续保？

预期：
- intent: [renewal_reminder]
- 走续保/通用分支
- 回复：说明续保流程、宽限期、保费是否会变化
```

### 测试 5：多轮对话 + 代词
```
第1轮输入：我的重疾险保额50万，缴费20年
（预期：系统记录 user_policy_type=重疾险, user_coverage_amount=50万）

第2轮输入：如果得了癌症能赔多少？
（预期：LLM_C1 结合对话历史，还原为"重疾险""50万保额""癌症=恶性肿瘤-重度在保障范围"）
```

### 测试 6：混合意图
```
输入：我的保单还有效吗？另外我查出高血压，这个影响保障吗？

预期：
- intent: [policy_lookup, coverage_inquiry]
- 走第一个匹配分支（保单查询）
- LLM_C3 判断出有未回答的保障咨询，在回复末尾主动询问高血压具体情况
```

## 6. 调试技巧

### 如果意图路由走错分支
1. 查看 LLM_C1 的输出，确认 intent 字段是否正确识别
2. 检查 If-Else 的条件顺序——`policy_lookup` 和 `coverage_inquiry` 如果同时出现，第一个匹配的生效
3. 调整 If-Else 的条件顺序，把更具体的意图放在前面

### 如果多轮对话丢失上下文
1. 检查 LLM_C1 和 LLM_C3 的"记忆"是否开启
2. 检查 Conversation Variables 是否已声明且 Prompt 中引用正确
3. 窗口大小建议 10 轮以上

### 如果某个分支无响应
1. 检查该分支的 KB 节点是否检索到内容（点击 KB 节点查看运行日志）
2. 检查 LLM_C2x 的上下文是否指向了正确的 KB 节点
3. 检查从 LLM_C2x → LLM_C3 的连线是否存在

## 7. 与模块A/B的联动（面试加分点）

```
模块A（产品配置） 输出的产品定义 JSON
  → 自动入库「保单服务知识库」中"保单条款与保障范围"文档
    → 模块C 的保障咨询可以直接检索到最新配置的产品条款

模块B（核保引擎） 输出的核保决策报告
  → 如果用户后续在模块C中问"为什么我的保费比别人贵"
    → 模块C 可以引用核保决策中的风险因子（职业/健康加费）来解释
```

面试时的说法：
> "三个模块不是孤立的——模块A配置产品规则，模块B用它核保，模块C用它服务。数据在三个模块间流转，形成'配置→核保→服务'的闭环。这在架构上就是 AI 中台的核心价值。"
