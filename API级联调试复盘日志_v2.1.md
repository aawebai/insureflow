## 关键教训

1. **低代码平台的能力边界**：Dify Cloud 的 HTTP Request 节点功能存在但沙箱不开放出站，这是平台文档中未明确说明的限制。涉及平台特有功能的方案，应先做最小可行性验证。

2. **无日志环境下的调试方法**：低代码平台不暴露执行日志时，控制变量法是唯一可靠的排查手段。三级逐层排除（基础结构→增量功能→变量隔离）系统性地锁定了根因。

3. **不固执于单一平台**：当平台功能限制无法绕过时，跳到外部工具解决是合理的技术选型，不丢分。关键是把排查过程讲清楚。

4. **"能导入"≠"能运行"**：Dify YML 文件的格式验证通过只代表配置合法，不代表功能在目标平台上可用。这是低代码平台特有的"假阳性"问题。
